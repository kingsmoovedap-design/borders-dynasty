# **DYNASTY OS — SYSTEM INTEGRATION BREAKDOWN**  
## *How the Load Board, Dispatch Engine, Driver App, and Website Work Together*

This document explains how the four core components of the Dynasty OS interact to form a unified logistics operating system.

# **1. SYSTEM OVERVIEW**

The Dynasty OS is built on four interconnected components:

1. **Load Board (Shipper + Ops View)**  
2. **Dispatch Engine (AI + Human Hybrid)**  
3. **Driver App (Execution Layer)**  
4. **Website / Frontend Console (User Interface Layer)**  

These components communicate through:

- **Dynasty OS API**  
- **Codex Event Logging**  
- **Treasury & Compliance Services**  

Together, they form a **closed operational loop**:

```
Load → Dispatch → Driver → Delivery → Treasury → Codex → AI → Load
```

---

# **2. LOAD BOARD (SHIPPERS + OPS)**

### **Purpose**
The Load Board is where loads are created, viewed, filtered, and managed.

### **Users**
- Shippers  
- Operations staff  
- Dispatchers  

### **Key Functions**
- Create new loads  
- View available loads  
- Filter by mode, region, status  
- View load details  
- See compliance flags  
- See risk score  
- Trigger dispatch suggestions  

### **Data Flow**
1. Shipper creates a load → `/loads`  
2. Load appears on the Load Board  
3. Ops can request AI dispatch suggestions  
4. All actions logged in Codex  

### **Outputs**
- Load metadata  
- Load status  
- Dispatch readiness  
- Compliance indicators  
- Codex events  

---

# **3. DISPATCH ENGINE (AI + HUMAN)**

### **Purpose**
Assign the right driver/courier to the right load at the right time.

### **Users**
- Dispatchers  
- Ops  
- AI engine  

### **Key Functions**
- AI driver scoring  
- Driver eligibility filtering  
- Backup driver suggestions  
- Manual override  
- Assignment confirmation  

### **Dispatch Flow**
1. Load Board triggers **AI Suggestion** → `/loads/{id}/suggest-dispatch`  
2. AI returns:
   - primary driver  
   - backup drivers  
   - score breakdown  
3. Dispatcher accepts or overrides  
4. Assignment sent to Driver App  
5. Codex logs:
   - DISPATCH_SUGGESTED  
   - DISPATCH_ASSIGNED  
   - DISPATCH_OVERRIDE  

### **Outputs**
- Assigned driver  
- Dispatch timeline  
- Risk-adjusted decisions  
- Codex events  

---

# **4. DRIVER APP (EXECUTION LAYER)**

### **Purpose**
Drivers and couriers execute the work.

### **Users**
- Drivers  
- Couriers  

### **Key Functions**
- View assigned loads  
- Update status (Arrived, Loaded, Delivered)  
- Upload POD  
- View earnings  
- Access credit  
- Track loyalty  

### **Driver Flow**
1. Driver receives assignment  
2. Driver updates milestones:
   - ARRIVED_PICKUP  
   - LOADED  
   - IN_TRANSIT  
   - ARRIVED_DELIVERY  
   - DELIVERED  
3. Driver uploads POD  
4. Treasury calculates payout  
5. Codex logs every event  

### **Outputs**
- Real-time status updates  
- POD documents  
- Driver performance data  
- Treasury triggers  
- Codex events  

---

### **Purpose**

# **6. HOW THEY WORK TOGETHER (FULL SYSTEM LOOP)**

Here is the **complete integration loop**, step by step.

---

## **STEP 1 — Load Created**
Shipper creates a load → Load Board updates → Codex logs `LOAD_CREATED`.

---

## **STEP 2 — Dispatch Triggered**
Ops requests AI suggestion → Dispatch Engine scores drivers → returns candidates.

---

## **STEP 3 — Driver Assigned**
Dispatcher accepts or overrides → assignment sent to Driver App → Codex logs.

---

## **STEP 4 — Driver Executes**
Driver updates milestones → POD uploaded → status updates flow to Load Board.

---

## **STEP 5 — Delivery Completed**
Load marked DELIVERED → Treasury calculates payout → Codex logs.

---

## **STEP 6 — Treasury & Credit**
Driver wallet updates → credit adjustments → rewards issued.

---

## **STEP 7 — Codex Anchoring**
All events stored → anchored → used for:
- audit  
- compliance  
- AI training  
- operational intelligence  

---

## **STEP 8 — AI Improves**
AI learns from:
- driver performance  
- lane history  
- risk patterns  
- compliance outcomes  

This improves future dispatch decisions.
